Background / 背景

Our platform is a B2B SaaS tool that helps product teams generate and manage technical documentation. The core AI pipeline handles five task types:

我们的平台是一个 B2B SaaS 工具，帮助产品团队生成和管理技术文档。核心 AI 流水线处理五种任务类型：

  1. Code-to-documentation / 代码转文档 (35% of calls) — Parse source code and generate API reference docs
  2. Content summarization / 内容摘要 (25%) — Condense long technical specs into executive summaries
  3. Translation / 翻译 (20%) — EN ↔ ZH ↔ JP technical translation

Routing Logic / 路由逻辑

// Simplified routing logic
function routeTask(task) {
  if (task.type === 'complex_analysis' || task.priority === 'critical') {
    return 'claude-opus-4';  // Tier 3 for the hard stuff
  }
  if (task.type === 'code_to_docs' && task.language === 'python') {
    return 'gpt-4o';  // Tier 2 — GPT-4o better at Python
  }
  if (task.type === 'translation' || task.type === 'summarization') {
    return 'deepseek-v3';  // Tier 1 — excellent for these tasks
  }
  return 'deepseek-v3';  // Default to cheapest
}

DeepSeek V3 handled 70% of our volume: summarization, translation, format conversion, and simple code-to-docs. GPT-4o covered 25%: complex language-specific code parsing. Claude Opus 4 handled only 5%: critical architecture reviews and security audits.

DeepSeek V3 处理了 70% 的工作量：摘要、翻译、格式转换和简单的代码文档化。GPT-4o 覆盖 25%：复杂语言的代码解析。Claude Opus 4 仅处理 5%：关键架构审查和安全审计。

What We Learned / 经验教训

1. Prompt Engineering Matters More with Cheaper Models / 便宜模型更需要 Prompt 工程

DeepSeek V3 needed more explicit instructions than GPT-4o. Where GPT-4o would "figure it out," V3 sometimes needed format examples. We invested 2 weeks in prompt optimization and saw V3's quality jump from 78% to 92% on our benchmark.

DeepSeek V3 比 GPT-4o 更需要明确的指令。GPT-4o 会"自己领会"的地方，V3 有时需要格式示例。我们花了 2 周优化 Prompt，V3 的基准测试质量从 78% 跃升至 92%。

2. Hybrid Routing Is Worth the Complexity / 混合路由的复杂性是值得的

Building the routing layer took about 3 engineering-weeks. But the $47K annual savings delivered a **60x ROI** on that investment in year one alone. The routing layer is now < 300 lines of code and easy to extend.

构建路由层大约花了 3 个工程师周。但每年 $47K 的节约在第一年就带来了 60 倍的投资回报。路由层代码不到 300 行，易于扩展。

3. Monitor, Then Migrate / 先监控，后迁移

Don't do a big-bang migration. We first logged 2 weeks of task data, analyzed which tasks could tolerate lower-tier models, then migrated incrementally — one task type per week. This gave us time to catch quality regressions before they affected users.

不要搞大爆炸式迁移。我们先记录了 2 周的任务数据，分析哪些任务可以容忍低层级模型，然后逐步迁移 — 每周一个任务类型。这给了我们时间在影响用户之前发现质量回归。

Should You Do This? / 你也该这样做吗？

If you're spending more than $500/month on AI APIs, the answer is almost certainly yes. The cost gap between flagship and economy models is now so large (50-100x) that running everything on the most expensive model is simply bad engineering economics.

如果你每月 AI API 支出超过 $500，答案几乎肯定是"应该"。旗舰模型和经济模型之间的成本差距如今如此巨大（50-100 倍），所有任务都用最贵模型跑，在工程经济上是不合理的。
